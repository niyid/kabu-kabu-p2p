Architecture Pattern

Client-Server vs Serverless P2P

A high-level comparison of two fundamental distributed system designs

Traditional
Client — Server
topology Hub-and-spoke — all traffic routes through one authority
data Stored centrally; server owns and controls it
trust required Users must trust the server operator
resilience low Server down = service down
scalability vertical Scale by upgrading the server
Serverless
Peer-to-Peer
topology Mesh — nodes connect directly to each other
data Distributed across peers; no central store
trust none needed Cryptographic identity, no operator
resilience high No single node can bring it down
scalability horizontal Capacity grows as peers join
Data Flow
Client-Server: A sends message to B
01Client A composes message
02A → encrypted request → Server
03Server decrypts, stores, re-encrypts
04Server → push/poll → Client B
05Server holds metadata & history
Serverless P2P: A sends message to B
01Peer A composes message
02A → end-to-end encrypted → Peer B
03No intermediary touches the payload
04Routed over overlay network (e.g. I2P)
05Only A & B hold the message
Tradeoffs
🔒
Privacy
Server sees metadata
No operator to surveil
⚡
Latency
Low, optimised routes
Higher — overlay hops
🛡️
Censorship
Easily blocked/shut down
Resistant by design
💸
Cost
Ongoing server/infra cost
Zero infra cost to operator
🔧
Complexity
Simpler client logic
NAT traversal, routing, churn
📴
Offline peers
Server buffers for you
Requires both peers online
