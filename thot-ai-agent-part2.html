Building an OS-level Android AI assistant that draws a floating panel above every app, reads whatever is on screen, and routes to five inference backends — from GPT-4o in the cloud to llama.cpp on-device. AI assistance shaped every layer of the process.

In Part I of this series, we dissected the XMRWallet codebase — examining how AI-assisted refactoring transforms a security-critical C++ JNI bridge, and what the niyid fork's systematic hardening reveals about working with AI tools in high-stakes software. That was a story about analysis: reading code, spotting patterns, understanding what changed and why.

This part is a story about construction. Thot is an Android application that draws a floating AI assistant over every app on the device. It reads whatever is visible on screen and puts a capable language model one tap away — while the user is in Gmail, WhatsApp, a banking app, or anywhere else. Building it required threading together Android's Accessibility API, the WindowManager overlay system, multiple LLM backends, and a clean architecture that keeps all of it testable and extensible. AI assistance shaped every layer of that process.

§ 01

What Thot Is — and Why It Is Hard to Build

The name Thot comes from the ancient Egyptian god of knowledge, writing, and wisdom — an appropriate patron for software that sits at the intersection of everything a user sees and thinks. The application's ambition is deceptively simple: place an AI assistant inside the Android operating system, not inside any single app.

That architectural choice — OS-level rather than app-level — is where the difficulty lives. A traditional AI chatbot runs in its own activity, with its own window, its own context. The user switches to it when they need it. Thot inverts this: it goes where the user already is, reads what is already on screen, and responds in place. The user never leaves their current context.

Implementing this requires three Android capabilities that rarely appear together in the same codebase:

The Accessibility API — designed for assistive technology, it grants read access to the view hierarchy of any app on the device. Thot uses it to extract the text visible on screen at any moment.

The WindowManager overlay system — the TYPE_APPLICATION_OVERLAY window type lets a service draw views above every other app. Thot uses it to render a draggable floating button and a response panel that appear over whatever the user is doing.

A multi-backend LLM routing layer — the user may be online or offline, on a fast device or a budget phone. Thot supports five inference backends: a cloud call to OpenAI, a cloud call to Anthropic, on-device inference via Google's MediaPipe and Gemma models, on-device inference via llama.cpp with any GGUF model, and a local rule-based fallback. The routing is seamless; the user picks a backend in settings and forgets about it.

§ 02

The AccessibilityService — Reading the World

ThotAccessibilityService is where Thot's OS integration begins. It listens for two event types across all applications: TYPE_WINDOW_STATE_CHANGED and TYPE_WINDOW_CONTENT_CHANGED. Every time the user navigates to a new screen or the content of the current screen updates, the service walks the view hierarchy from the root and collects all visible text.

The text extraction strategy

The extraction logic is recursive with two hard limits: a depth cap of 12 levels (deep enough to reach text in most apps, shallow enough to avoid runaway traversal) and a character cap of 3,000 (enough content to give an LLM meaningful context, small enough to keep prompt costs reasonable).

ThotAccessibilityService.kt — extractText() Kotlin
private fun extractText(node: AccessibilityNodeInfo, depth: Int = 0): String { if (depth > MAX_DEPTH) return "" // depth cap val sb = StringBuilder() val text = node.text?.toString() val contentDesc = node.contentDescription?.toString() if (!text.isNullOrBlank()) sb.append(text).append("\n") else if (!contentDesc.isNullOrBlank()) sb.append(contentDesc).append("\n") for (i in 0 until node.childCount) { val child = node.getChild(i) ?: continue sb.append(extractText(child, depth + 1)) if (sb.length > MAX_CHARS) break // early exit — avoid wasted traversal } return sb.toString().take(MAX_CHARS) }

The early exit on sb.length > MAX_CHARS inside the loop is notable. A naive implementation would build the entire string and then truncate it — wasting traversal time on nodes whose text will be discarded. The early exit was a specific AI suggestion during review, identified as a latent performance issue in the original draft.

Thread safety via @Volatile

The captured screen context is stored in a companion object as a @Volatile field. The overlay service and the accessibility service run on different threads — the accessibility service on the Android main thread, the overlay service potentially on a coroutine dispatcher. @Volatile ensures writes are immediately visible to reads without requiring a mutex for this single-writer, multiple-reader scenario.

💡
Design Note: self-exclusion guard

The service explicitly filters out its own package name (com.techducat.thot) from context capture. Without this guard, Thot would attempt to read and summarise its own settings screen — a recursive absurdity that produces useless prompts and wastes API calls.

§ 03

The Overlay System — Drawing Above Everything

ThotOverlayService is a foreground service that manages two WindowManager views: a floating action button and an action panel. This is the part of the codebase most directly shaped by Android's historical technical constraints, and the part where AI assistance was most valuable for navigating subtle timing and threading requirements.

The foreground service requirement

On Android 8 and above, a background service that draws overlays must be elevated to a foreground service — it must show a persistent notification to signal to the user (and to the OS) that it is intentionally active. Without this, the OS may kill the service to reclaim memory, causing the overlay to vanish unexpectedly. The notification is kept at minimum priority so it appears at the bottom of the shade without demanding attention.

The vsync timing problem

One of the more subtle issues that arose during development: calling WindowManager.addView() synchronously inside onCreate() produces intermittent "Already have a pending vsync event" warnings. The overlay occasionally failed to render on the first draw. The fix is to post the addFab() call to the main handler, allowing the current frame to commit before the new view is added.

ThotOverlayService.kt — vsync-safe FAB init Kotlin
override fun onCreate() { super.onCreate() // ... // Post to run after the current frame is committed — // avoids "pending vsync" warnings from WindowManager. Handler(Looper.getMainLooper()).post { addFab() } }

This is the kind of fix that is difficult to find by reading documentation. The vsync warning is easy to dismiss as noise; the rendering failure is intermittent enough to miss in casual testing. An AI assistant, having encountered this pattern across many Android codebases, flagged it during the code review pass and produced the fix in the same response.

The keyboard focus problem

Overlay windows default to FLAG_NOT_FOCUSABLE — they do not intercept touch events intended for the underlying app. This is correct for the FAB, which should not steal focus from the user's app. But when the user taps the custom task text field in the panel, the soft keyboard must appear — which requires removing FLAG_NOT_FOCUSABLE from the panel's WindowManager.LayoutParams at that moment, and restoring it when the keyboard is dismissed.

ThotOverlayService.kt — keyboard focus toggle Kotlin
private fun allowKeyboard() { val pp = panelParams ?: return pp.flags = pp.flags and WindowManager.LayoutParams.FLAG_NOT_FOCUSABLE.inv() try { windowManager.updateViewLayout(panelView, pp) } catch (_: Exception) {} } private fun denyKeyboard() { val pp = panelParams ?: return pp.flags = pp.flags or WindowManager.LayoutParams.FLAG_NOT_FOCUSABLE try { windowManager.updateViewLayout(panelView, pp) } catch (_: Exception) {} }

The bitwise flag manipulation — .inv() to clear a specific flag — is idiomatic Android but easy to get wrong. The AI draft on first attempt used a direct assignment that overwrote all flags rather than toggling only the focusability bit, which would have broken the overlay's touch-passthrough behaviour. The corrected pattern above was produced after the error was explained.

The FAB edge-snapping behaviour

The floating button snaps to the left or right edge of the screen when the user lifts their finger after dragging. The snap decision is based on whether the button's centre position is left or right of the screen midpoint. The snap position is persisted to SharedPreferences so the button reappears in the same location on next launch.

§ 04

The Routing Layer — Five Backends, One Interface

ThotCoreProvider is the central dispatcher. It holds lazy references to all five provider implementations and routes a ThotTask to the correct one based on the ProviderType enum attached to the task.

The routing switch is deliberately explicit rather than clever. Each case is named after the provider it dispatches to. There is no reflection, no plugin registry, no factory class. The entire routing logic is visible in a single when expression.

ThotCoreProvider.kt — runTask() Kotlin
override fun runTask(task: ThotTask, callback: (String) -> Unit) { when (task.provider) { ProviderType.LOCAL -> scope.launch { val result = withContext(Dispatchers.IO) { localProvider.handle(task.buildFullPrompt()) } callback(result) } ProviderType.MEDIAPIPE -> scope.launch { val result = withContext(Dispatchers.IO) { mediaPipeProvider.handle(task.buildFullPrompt(), prefs.localModelPath) } callback(result) } ProviderType.LLAMACPP -> scope.launch { val result = withContext(Dispatchers.IO) { llamaCppProvider.handle(task.buildFullPrompt(), prefs.localModelPath) } callback(result) } ProviderType.OPENAI -> openAiProvider.query(task.buildFullPrompt(), callback) ProviderType.ANTHROPIC -> anthropicProvider.query(task.buildFullPrompt(), callback) } }

Why the callback pattern, not suspend functions

The API clients (OpenAI, Anthropic) use OkHttp's async enqueue() and deliver results on the main thread via a Handler. The on-device providers are synchronous and run on Dispatchers.IO. Both paths ultimately deliver a String to the same callback. This callback-based interface was a deliberate choice over suspend functions: the overlay service is a plain Android Service that does not own a coroutine scope. The coroutine scope lives inside ThotCoreProvider, which cancels it cleanly in onDestroy().

§ 05

The Anthropic Client — Claude as Thot's Voice

The AnthropicClient wraps the Anthropic Messages API using OkHttp. It is a self-contained class with no external dependencies beyond the HTTP client and a reference to ThotPreferences for the API key. The system prompt embedded in the client is where Thot's persona is defined:

"You are Thot, a helpful AI assistant embedded inside the Android operating system. You can see the user's current screen context. Be concise, practical, and actionable. When given screen context, reference it directly. Format your responses for a mobile display — short paragraphs, avoid lengthy preambles."

— AnthropicClient.kt, SYSTEM_PROMPT constant

The instruction to format for mobile display is not cosmetic. A response appropriate for a desktop chat interface — with headings, bullet lists, and multiple paragraphs — is illegible in a 300dp overlay panel on a phone screen. The system prompt constrains the model to produce responses that are readable in context.

The prompt construction pipeline

Before any API call is made, the task's buildFullPrompt() method assembles the final prompt from two pieces: the screen context captured by the accessibility service, and the user's actual request.

ThotTask.kt — buildFullPrompt() Kotlin
fun buildFullPrompt(): String = if (screenContext.isBlank()) { prompt } else { """ |[Screen context captured from current app] |$screenContext | |[User request] |$prompt """.trimMargin() }

The section labels — [Screen context captured from current app] and [User request] — are deliberate. They help the model distinguish between information it is being shown about the environment and the instruction it is being asked to follow. Without the separation, models occasionally conflate the two and attempt to "explain" the user's own request rather than responding to it.

This prompt structure was refined through several iterations with the AI assistant acting as both author and critic — writing a prompt version, then evaluating its own output against sample screen contexts to identify where the model's response missed the mark.

§ 06

The ChronoManager — Scheduled Automations

Thot includes a lightweight automation engine — ChronoManager — that allows developers embedding Thot to schedule arbitrary coroutine-based tasks with either a one-shot delay or a repeating interval. This is the most forward-looking part of the codebase: a foundation for building agents that act on the user's behalf rather than waiting to be asked.

ChronoManager — usage example Kotlin
val manager = ChronoManager() // One-shot: send an email 10 minutes from now manager.addScript( ChronoScript(id = "welcome_email", description = "Send welcome email") { EmailAction(context).sendEmail("user@example.com", "Hello", "Welcome!") }, delayMillis = 10 * 60 * 1000L ) // Repeating: check something every hour manager.addRepeatingScript( ChronoScript(id = "hourly_check", description = "Hourly task") { // suspend work runs here }, intervalMillis = 60 * 60 * 1000L ) // Cancel cleanly from onDestroy manager.cancelAll()

The EmailAction and SmsAction classes are intentionally thin — they launch the system email chooser and SMS app respectively, rather than sending silently in the background. This is a privacy boundary: Thot can surface a communication intent on the user's behalf, but the user always confirms the send. Silent background communication would require explicit opt-in and far more complex permission handling.

Firebase dependencies are scoped to the playstore flavor via "playstoreImplementation"(...). The F-Droid build has no GMS dependency whatsoever — no Firebase, no Play Services, no tracking. The IS_FDROID_BUILD boolean build config field gates any analytics or crash reporting calls at compile time, not at runtime.

API keys are read from local.properties or from CI environment variables, never from the application itself. This is the same secrets pattern employed in the XMRWallet fork reviewed in Part I: credentials that live in version control are credentials that will eventually be compromised.

§ 08

Security Considerations

Thot's threat model is different from XMRWallet's — it handles no cryptographic key material, manages no funds, and performs no blockchain operations. But its access to the device is more expansive than almost any other application: it can read the screen content of every other installed app. That privilege deserves examination.

⚠️
Risk: Screen content privacy

When using a cloud backend (OpenAI or Anthropic), the text visible on the user's screen — which may include passwords, banking information, private messages, or medical data — is transmitted to a third-party API. The user must understand this before enabling a cloud provider. On-device providers (MediaPipe, llama.cpp, LOCAL) never transmit screen content outside the device.

🛡️
Mitigation: Package filtering

The accessibility service's onAccessibilityEvent handler explicitly skips events from Thot's own package. A future hardening step would extend this to skip known password manager packages and system keychain activities where screen content capture is clearly inappropriate.

📋
Note: Permission transparency

The prominent disclosure dialog displayed on first launch explains precisely what the Accessibility permission enables and why it is needed. This is both a Play Store requirement and a design principle: users who understand what they are granting make more informed decisions about which provider to select.

§ 09

The AI Fingerprint in Thot

The question that animates both parts of this series is: what does AI-assisted development actually look like in practice? Not in theory, not in marketing — in the code itself. Thot offers a different specimen than the XMRWallet fork. Where the fork was AI-assisted refactoring of an existing codebase, Thot is AI-assisted construction of a new one. The patterns are different.

Architecture clarity from early critique. The initial design put the LLM routing logic inside ThotOverlayService. The AI assistant flagged this as a violation of single responsibility: the overlay service manages views and user interaction; the routing logic belongs in a dedicated provider. The separation into ThotCoreProvider was a direct consequence of that critique — and it is the decision that makes the codebase testable.

Defensive patterns at the system boundary. The permission check before addFab(), the post-to-main-handler timing fix, the bitwise flag toggling for keyboard focus — all three were produced during review passes where the AI assistant read the draft code and identified failure modes that are platform-specific and not obvious from documentation alone. The pattern is identical to what we saw in the XMRWallet fork: AI tools excel at the intersection of "well-documented best practice" and "routinely missed in human-written code."

Prompt engineering as a first-class concern. The structured prompt format in ThotTask.buildFullPrompt() — with explicit section labels separating context from instruction — was refined iteratively. The AI assistant both wrote candidate prompt templates and critiqued its own responses to them, identifying where model confusion arose from ambiguous prompt structure. This self-evaluation loop is a development pattern that only makes sense when the thing being built is itself an AI product.

What was not AI-generated. The decision to support five providers rather than one — the architectural bet that on-device inference via llama.cpp is worth the complexity of an NDK submodule — was a human call. The privacy framing of the permission model, the choice to launch a system email chooser rather than send silently, the F-Droid distribution flavor: these are product values, not code patterns. AI tools operate within a brief; the brief itself is the human contribution that matters most.

§ 10

Extending Thot — The Developer API

Thot is designed to be embedded in other applications, not only used as a standalone app. The BaseThotActivity base class exposes a single method — requestLLMContext() — that dispatches a task to the configured provider and delivers the result on the main thread.

BaseThotActivity — embedding Thot in another app Kotlin
class MyActivity : BaseThotActivity() { override fun onCreate(savedInstanceState: Bundle?) { super.onCreate(savedInstanceState) val task = ThotTask( prompt = "Summarise the payment flow on this screen.", screenContext = "Checkout — Total: ₦45,000 — Pay with card", provider = ProviderType.ANTHROPIC ) requestLLMContext(task) { response -> // delivered on main thread — update UI directly myTextView.text = response } } }

The screenContext parameter does not have to come from the Accessibility API. Any string can be passed — a database query result, a product description, a transaction receipt. The accessibility integration is one source of context, not the only one. This flexibility is what makes Thot a platform rather than a point solution.

Conclusion

The Brief Is the Human Contribution That Matters Most

Thot and the XMRWallet fork share a common characteristic despite being very different projects: in both cases, the most consequential AI contributions were not the ones that wrote the most code. They were the ones that identified the most important problems — a missing vsync timing guard, a misconfigured window flag, an architectural boundary that needed to be drawn — before those problems reached production.

The limiting factor in AI-assisted development is not the AI's ability to write code. Any capable model can produce syntactically correct Kotlin or idiomatic C++. The limiting factor is the quality of the brief: the clarity of the problem statement, the depth of the platform knowledge brought to the review, and the willingness to treat the AI's critiques as real engineering input rather than suggestions to be politely ignored.

Thot works. It draws a floating button over every app, reads the screen, routes to the right model, and delivers a response in a panel that appears in context where the user already is. For a project of this architectural complexity — crossing OS-level APIs, JNI inference runtimes, and cloud services in a single codebase — the velocity that AI assistance enabled was measurable. So were the bugs it caught.

The Egyptian god Thot was the keeper of records, the documenter of what was true. The software that bears his name is, in a smaller way, trying to do the same thing: take what is visible on screen and make it more legible, more actionable, more useful. Whether the AI assistance used to build it qualifies as intelligence is a question for another article. That it helped build something better is not in doubt.
